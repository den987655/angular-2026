
**ГЛАВА 1.**  
Эволюция фронтенда: от SPA и Redux к Signals и Server-First

Фронтенд вырос в самостоятельную инженерную дисциплину молниеносно, но большинство проектов до сих пор несут на себе груз решений десятилетней давности: огромные SPA, глобальный Redux/NgRx, структура по типам файлов, «компоненты-боги».

Чтобы понять, почему сейчас всё движется к signals, server-first и фича-ориентированным срезам, пройдём путь эволюции — коротко, без ностальгии.

1. Эпоха jQuery (до ~2014)  
   «Добавить немного динамики».  
   HTML верстался вручную, JavaScript был протезом.  
   Логика разбросана, прямой доступ к DOM, никакой реактивности.  
   Архитектуры не существовало → и спорить было не о чем.

2. Эпоха больших SPA (2014–2018)  
   AngularJS, React, Vue, Ember.  
   Появилось ощущение: «Мы можем построить настоящее приложение в браузере».  
   Плюсы были очевидны: компоненты, роутинг, единый state.  
   Минусы проявились позже: толстый клиент, гигантские бандлы, мёртвый SEO, медленный первый рендер, хаос в управлении состоянием → родился Redux/NgRx как симптом болезни.

3. Эпоха «глобального Store» и архитектур ради архитектуры (2017–2020)  
   Команды пытались побороть хаос централизацией.  
   Redux, NgRx, Vuex, MobX, Akita — всё это было попыткой удержать взрывающуюся сложность.  
   Результат: boilerplate, оверхед, медленный dev loop, «божественный Store», в который тащили вообще всё.  
   Главный урок эпохи: слепое копирование бэкенд-паттернов (Clean Architecture, Onion) на фронте разрушает так же эффективно, как их отсутствие.

4. Эпоха SSR и server-first (2020–2024)  
   Индустрия поняла: толстый клиент — это зло.  
   Появились Next.js, Remix, Nuxt 3, Angular Universal/Hydration, React Server Components, islands architecture, streaming SSR.  
   Главная идея: «Отдаём готовый HTML сразу, а JavaScript — только то, что действительно нужно для интерактивности, и то по частям».

5. Эпоха fine-grained реактивности (2023–2026+)  
   Angular Signals, SolidStart, Vue Vapor, Svelte 5, Qwik, Preact Signals — все крупные игроки перешли на сигналы.  
   Почему это переворот:  
   - минимум перерендеров,  
   - предсказуемое поведение,  
   - проще reasoning,  
   - отсутствие «магии» RxJS/Zone.js,  
   - идеальная совместимость с SSR и частичной гидрацией.  

   Сигналы фактически решили те самые проблемы, ради которых когда-то изобретали глобальные стоны.

6. Эпоха сближения и доминирования фича-срезов (2024 → сейчас)  
   Фреймворки сходятся:  
   - Angular, React, Vue теперь рекомендуют одну и ту же структуру: фичи → страницы → ui → state → data-access.  
   - Server-first стирает грань между фронтом и бэком.  
   - Signals делают клиентский код лёгким и предсказуемым.  
   - Фича-ориентированный подход стал стандартом де-факто в крупных проектах.

   Архитектура перестала быть «структурой папок».  
   Это теперь система уровней ответственности и потоков данных.

Почему именно фичи победили

Потому что все предыдущие подходы сломались на масштабе:

- структура по типам → теряется контекст фичи,  
- огромный SPA → дорого поддерживать,  
- глобальный Store → чёрный ящик,  
- SSR/hydration → требует чёткого разделения ответственности,  
- signals → поощряют локальность состояния.

Итог: vertical slices (фича-ориентированные срезы) — это не мода, а единственный подход, который выдерживает реальную сложность 2026+.

Пример современной структуры:

```
/features
  /catalog
    /ui
    /components
    /pages
    /model
    /state
    /data-access
    /services
  /order
  /cart
  /profile
```

**Итог главы 1**

Сигнал  
Толстые SPA мертвы. Их убили Web Vitals, мобильный трафик и SEO.

Ключевая парадигма 2026+  
Server-first + fine-grained реактивность + фича-ориентированные вертикальные срезы.

Системные последствия  
- Структура по типам больше не масштабируется.  
- Глобальный Store — исключение, а не правило.  
- SSR/hydration — обязательная часть архитектуры, а не плагин.  
- Фича становится основной единицей проектирования и код-ревью.

Практические шаги прямо сейчас  
1. Начинайте думать фичами, а не типами файлов.  
2. Выносите тяжёлую логику из компонентов.  
3. Локальный UI-state оставляйте в UI, доменный — в фича-сторах.  
4. Проектируйте проект под SSR/hydration с первого дня.

Риски  
- Попытка «перепилить всё сразу» → смерть проекта.  
- Фича-папки без строгой дисциплины быстро превращаются в свалку.

Что следить в 2026–2027
- Angular Signals & Hydration improvements  
- React Server Components + Actions стабилизация  
- Vue Vapor mode  
- Qwik/Resumability эволюция  
- Новые инструменты частичной гидрации и streaming
